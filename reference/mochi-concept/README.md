## 待验证问题（可证伪假设）

> **如果** 玩家完成一个完整的微循环——拉摇杆 → 粉碎 → 剪影 → 点击揭晓（约 30 秒），**他们会感受到一种小剧场式的满足感。**
> **确认条件：** 3 名以上首次试玩者在首次揭晓后 **10 秒内** 自发地再次拉动摇杆，无需任何提示。

最高风险假设：拉摇杆 → 震动 → 揭晓的微剧场，必须仅靠 **占位视觉 + 合成音效** 就让人感到内在满足。如果做不到，后期再多 iOS Haptic 调校也救不了这个概念。

## 如何玩

1. **IDLE** 状态——拖动右侧的 **橙色把手** 向下
2. 至少拉到 80% 行程才会 **触发粉碎**；短拉会回弹（测试阻尼手感）
3. **CRUSHING**——机器震动 1.4 秒，碎屑翻滚，内部闪光，伴随低沉轰鸣和撞击音效
4. **REVEALING**——黑色剪影从底部托盘弹出，伴随 "ding" 音
5. **点击剪影**——它会弹跳并被随机颜色填充
6. 约 1.4 秒后回到 IDLE。再次拉动。

**状态栏会显示当前状态和累计拉动次数。**
**每当你在揭晓后再次拉动时，控制台会打印 `[METRIC] seconds_to_next_pull=X.X`**——这是假设验证的关键信号。

## 本原型验证什么

✓ 摇杆拉动手感（鼠标拖拽 + 弹簧回弹）
✓ 粉碎动画节奏（震动幅度、时长、碎屑）
✓ 剪影 → 点击 → 填色的揭晓微剧场
✓ 音频支撑（合成的 click / thunk / rumble / ding——无需任何真实音效素材）
✓ "我还想再拉一次吗？"——核心留存信号

## 本原型不验证什么

✗ iOS Haptic 反馈——桌面没有 Taptic Engine。**需要独立的原型 2**。
✗ 文本输入 UX——延后（不影响剧场假设）
✗ 货架 / 收藏——延后（长期留存范畴，不属于微循环）
✗ 真实美术（仅用 debug 矩形）
✗ 真实音频（合成的占位音；最终音效需按 game-concept.md 进行专业设计）

## 调参旋钮（手感不对时可调）

在 `main.gd` 中：

| 常量 | 默认值 | 作用 |
|------|--------|------|
| `LEVER_TRIGGER_Y` | 180.0 | 摇杆需拉多远才触发粉碎。值越小越容易触发（投入感越弱）。 |
| `LEVER_PULL_MAX` | 220.0 | 最大拉动距离。 |
| `_shake_machine` `amp` | 8.0 | 震动幅度（像素）。越大越剧烈。 |
| `_shake_machine` `freq` | 28.0 | 震动频率（每秒）。 |
| `_shake_machine` `dur` | 1.2 | 震动时长（秒）。 |
| `_enter_crushing` timer | 1.4 | 揭晓前的总粉碎时长。 |
| `_enter_complete` timer | 1.4 | 产品停留至重置的时长。 |
| `_make_tone "thunk"` freq | 180.0 | 摇杆拉到底时的低频撞击声。可尝试 120–240。 |
| `PRODUCT_COLORS` / `PRODUCT_SHAPES` | — | 增加或修改变体。 |

**剧场节奏检查：** 整个微循环时长应按 game-concept.md 控制在 **22–30 秒**。当前 *机器* 部分约 3.0 秒（粉碎 1.4 + 揭晓弹出 0.45 + 你的点击 → 停留 1.4）。在正式游戏中需额外加入约 5–15 秒玩家"写烦恼"的时间。

## 试玩协议

把这个原型交给某人（或在间隔 2 天以上后给自己）：

1. **只说一次开场白：** "拖动右侧的橙色把手向下。" 其他什么都不说。
2. **退后一步。不要旁白。不要协助。**
3. **静静观察。** 记录每一次犹豫、回看、困惑的瞬间。
4. 在他们停下或拉到第 5 次时，**只问一个问题**：
   > "刚才哪里让你感到困惑或不满足？"
5. 记下他们是否 **未经提示** 自发拉了第二次——以及从首次揭晓到第二次拉动之间过了多少秒。

> 只有 **3 名以上首次试玩者** 在 10 秒内自发再次拉动，假设才算被确认。三个人不算多——去找到三个人。

### 结论

微循环的 **逻辑** 站得住——拉 → 粉碎 → 揭晓的节奏读起来像一场迷你剧场。最高风险假设（无 Haptic 时的剧场质量）没有破——iOS Haptic 仍然是抛光层，而非结构性支柱。

**关键警示（用户 2026-05-21）：** 视觉 / 交互层面 **过于寡淡**。循环在机制上是通的，但还没能以概念所承诺的强度把剧场感卖出去。这 **不是** "之后再抛光" 类型的反馈——意味着 **Game Feel / Juice / VFX 必须被当作一等系统对待**，而不是收尾工序。

带入 Pre-Production 的具体启示：
- 挤压拉伸、预备动作、跟随动作必须从一开始就设计进摇杆 / 机器 / 剪影 / 揭晓的动作里——不能最后再加。
- 粒子、屏幕震动、冲击时的时间缩放、颜色闪烁、音频分层应归入专门的系统 GDD（`game-feel.md` 或同等文档）。
- juice / VFX 系统大概率需要在 vertical slice 之前自带一道 gate——若它无法让循环 *感觉上具备剧场强度*，需要 pivot 的是概念本身，而不仅仅是实现。

### 暴露的 bug（沉淀为生产编码规范）

Godot 4.6 更严格的类型推导拒绝了三种原型级写法：
- `var x := clamp(...)`——泛型 `clamp` 返回 `Variant`。应使用 `clampf`。
- 变量名为 `wrap`——覆盖了内置函数。避免使用 `wrap / abs / sign / max / min / clamp / lerp` 这类名字作为局部变量。
- 带类型标注的变量从辅助函数赋值——标注必须与辅助函数的实际返回类型匹配，而不是与意图类型匹配。

这些将作为 `godot-gdscript-specialist` code review 的 checklist 条目。
